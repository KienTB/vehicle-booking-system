# CARENTO SEAMLESS PREMIUM UI GENERATION SKILL

## Primary Objective

Generate production-ready Carento web interfaces that feel:

* premium,
* seamless,
* integrated,
* calm,
* refined,
* visually restrained,
* conversion-friendly.

The generated UI must not feel like:

* stacked landing page sections,
* box-after-box premium cards,
* marketing module compositions,
* enterprise admin dashboards,
* visually fragmented AI-generated blocks.

Carento pages must read as one cohesive interface, not a collection of separate visual chunks.

---

## GLOBAL RENDERING PHILOSOPHY

The generator must prioritize:

1. seamless visual flow
2. typography hierarchy
3. alignment discipline
4. negative space
5. restrained premium emphasis
6. conversion clarity
7. implementation cleanliness

Visual attractiveness must come from restraint, not from adding more blocks.

Premium does not mean more containers.
Premium means fewer but stronger visual decisions.

---

## CORE LAYOUT PHILOSOPHY

### 1. Integrated Page Composition

Pages should be rendered as one continuous composition.

Avoid treating every information group as an isolated card.

The page should visually flow through:

* spacing,
* typography,
* subtle separators,
* image anchors,
* alignment,
* controlled surface contrast.

Use hard card containers only where functionally necessary:

* forms,
* booking summary,
* modal,
* filter shell,
* data tables when needed.

Everything else should feel embedded into the page canvas.

---

### 2. Minimal Surface Blocking

Do not create:

* hero block,
* trust block,
* CTA block,
* support block,
* promo block

as visibly separated giant modules unless the page truly requires it.

Premium Carento UI should avoid the “AI assembled section stack” look.

Use background shifts sparingly.
Use containers sparingly.
Use cards sparingly.

---

### 3. Visual Hierarchy by Typography, not by Boxes

Hierarchy should come from:

* heading scale,
* spacing rhythm,
* text weight,
* image proportion,
* CTA contrast,
* alignment.

Not from endlessly wrapping content into elevated panels.

---

### 4. One Primary Focus Rule

Each screen should have one clear visual focus only:

* one dominant form,
* or one dominant vehicle image,
* or one dominant booking action.

Do not create multiple competing visual anchors.

---

## CONTAINER USAGE RULE

Allowed major containers only when needed:

* auth form shell,
* booking summary panel,
* vehicle filter bar,
* modal/dialog,
* table/list wrapper.

Disallowed behavior:

* wrapping every trust note in cards,
* wrapping every text section in floating panels,
* creating nested premium boxes inside premium boxes.

The page must breathe through whitespace, not through card spam.

---

## AUTH PAGE GENERATION RULE

Login/Register/Forgot Password must feel:

* refined,
* clean,
* high-confidence,
* calm.

Preferred:

* balanced split composition or centered integrated shell,
* one strong vehicle/brand visual anchor,
* one form anchor,
* minimal secondary notes.

Do not create:

* multiple trust cards,
* extra support boxes,
* floating decorative chips,
* testimonial clutter.

The auth page should feel like one premium screen, not left block + right block + extra mini blocks.

---

## LISTING PAGE GENERATION RULE

Vehicle listing pages should feel fluid and scannable.

Preferred:

* integrated top filter line,
* seamless card grid,
* minimal interruption between browsing areas.

Do not create separate giant showcase sections above the listing unless explicitly requested.

Browsing flow must feel continuous.

---

## DETAIL PAGE GENERATION RULE

Vehicle detail page should revolve around:

* large vehicle imagery,
* clean information hierarchy,
* booking action clarity.

Specs, policies, trust notes, and price should feel naturally aligned into the page — not as 8 different cards.

Use grouped spacing and subtle dividers before using separate containers.

---

## BOOKING PAGE GENERATION RULE

Booking pages must reduce anxiety through clarity, not through excessive visual grouping.

Use:

* progress stepper,
* integrated grouped fields,
* one booking summary panel,
* one confirm CTA.

Avoid giant repeated boxed form groups.

---

## ACCOUNT PAGE GENERATION RULE

Account pages should feel like a refined service dashboard.

Not enterprise.
Not landing page.

Use:

* simple grouped history,
* aligned account controls,
* restrained card usage.

---

## CTA RULE

Each page should have:

* one dominant CTA,
* minimal secondary CTA.

Do not scatter many colored action buttons around the page.

CTA visibility should come from:

* color contrast,
* placement,
* whitespace,
* size.

Not from wrapping CTA into another container.

---

## IMAGE USAGE RULE

Vehicle imagery should act as premium anchors.

Use:

* one strong visual area,
* clean consistent supporting thumbnails if needed.

Do not create multiple decorative image banners just to fill sections.

---

## COMPONENT STATE RULE

All interactive components still must define:

* default
* hover
* focus-visible
* active
* disabled
* loading
* error

But state polish should remain subtle.

No exaggerated hover gimmicks.

---

## RESPONSIVE RULE

Mobile must feel like a compact premium app screen.

Not stacked desktop sections.

When collapsing:

* merge surfaces,
* reduce visual fragmentation,
* keep one continuous reading flow.

Avoid mobile feeling like 10 cards stacked vertically.

---

## FORBIDDEN RENDERING PATTERNS

Never generate:

* landing page section stack behavior,
* hero block + trust block + CTA block + note block composition,
* premium card spam,
* excessive floating surfaces,
* fragmented AI-generated module feeling,
* decorative content boxes without functional reason.

If the page feels assembled from chunks, regenerate cleaner.

---

## CONTENT TONE

Tone must remain:

* concise,
* premium,
* reassuring,
* customer-friendly,
* booking-focused.

But copy quantity should be restrained.

Do not fill UI with unnecessary support sentences.

---

## FINAL SELF-QA

Before output verify:

* Does the page feel visually seamless?
* Does it avoid obvious block stacking?
* Is the page using fewer containers?
* Is hierarchy coming from typography/spacing rather than cards?
* Does it feel premium through restraint?
* Does it feel like one composed product screen?

If not, simplify further before output.
